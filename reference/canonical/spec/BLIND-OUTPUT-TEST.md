# BLIND-OUTPUT-TEST.md — The Human Verification Protocol

> **Canonical spec.** The ultimate test of whether an output is creator-grade is whether someone who knows the creator's work can tell it apart from creator-written. Three evaluators. Mixed samples. Single-blinded. The Blind Output Test is the operational definition of "encoding is complete."

---

## The principle

Triple-Layer Verification (formal + semantic + info-theoretic) catches ~95% of bad output. The remaining 5% are outputs that LOOK fine to mechanical/semantic/compression checks but are subtly wrong in ways only a human reader who knows the creator can detect.

The Blind Output Test catches that 5%.

It is the **human-grade pass/fail gate** that determines whether the OS's encoding has reached the **Elite tier** (90%+ context, output blind-test indistinguishable from creator's own).

Without the Blind Output Test, the encoding can drift — the OS could converge on outputs that score well on automated gates but feel "off" to anyone who actually knows the creator. The Blind Output Test is the truth-anchor.

---

## When to run

The Blind Output Test runs:
1. **At graduation** — when a skill is being promoted from draft to production for the first time
2. **Per-creator calibration** — once a skill is encoded with the creator's specific compartments, run the test before declaring the skill "creator-ready"
3. **Drift detection** — quarterly, on randomly sampled outputs, to catch encoding drift
4. **After sacred-format changes** — when SYSTEM.md, INVARIANTS.md, ENCODING.md, or company.yaml change in non-trivial ways
5. **After model upgrades** — when the underlying LLM is swapped (e.g., Claude 4.6 → 4.7), to verify the encoding holds

Not every skill output runs through Blind Output Test. The Test has overhead. It's a gating event, not a per-execution check.

## The protocol

### 1. Sample preparation

For each Test event:
- 3 samples generated by the OS using the skill being tested
- 3 samples written directly by the creator (or a writer the creator considers voice-accurate)
- Total: 6 samples
- Same prompt / same brief / same target across all 6
- All samples stripped of any metadata that would identify the source (no Signal 5-tuple footer, no compartment citation, no "Confidence: HIGH" tag — those are meta-data tells)

### 2. Evaluator selection

3 evaluators required. Each must:
- Have spent ≥ 10 hours consuming the creator's content (videos, posts, emails, sales calls — wherever the creator's voice lives)
- Be capable of articulating "this sounds like the creator" or "this doesn't" with reasoning
- Not have been involved in producing any of the samples

Evaluators can be:
- The creator's audience members (top commenters, long-time subscribers)
- The creator's own team (if they spend significant time consuming creator output)
- The creator's customers / mentees / friends-of-the-channel
- External evaluators recruited specifically (paid panel)

The evaluators should NOT all be from the same source category (e.g., not 3 audience members and no team — variance helps).

### 3. The blind read

For each sample, each evaluator answers:
- "Did the creator write this, or the system?" (binary)
- "How confident are you?" (1–5)
- "What signal made you think so?" (open text — voice / pacing / a specific phrase / topic / structure / etc.)

Order randomized. Evaluators don't know how many of the 6 are creator vs. system. They evaluate each sample independently.

### 4. Scoring

Per sample:
- 3 evaluator votes
- "Creator" = vote for creator-written; "System" = vote for system-written

Test verdict:
- **PASS** — at least one of the system samples receives ≥ 1 "Creator" vote out of 3 evaluators
- **FAIL** — all system samples receive 0 "Creator" votes (i.e., 3/3 evaluators correctly identify all 3 system samples as system-written)

Alternative formulations:
- **STRONG PASS** — ≥ 2/3 evaluators voted "Creator" on at least one system sample (the encoding is strong enough to fool informed readers)
- **WEAK PASS** — exactly 1/3 voted "Creator" on a system sample (the encoding is competitive but identifiable)

The PASS threshold is deliberately lenient. The signal isn't "fool everyone." The signal is "produce output that can plausibly come from the creator." If even one informed evaluator believes a system sample is creator-written, the encoding has cleared the human-grade bar.

### 5. The "what signal made you think so" log

The open-text reasoning is the most valuable artifact of the Test. Even on a PASS, the reasoning surfaces:
- Specific phrases / cadences the evaluators key on as creator-typical
- Structural patterns that read as creator-typical
- Failure modes that pull a sample toward "system-written" perception (these become candidates for `phrases_to_avoid` or for more aggressive voice-fidelity training)

Every Test event's reasoning log gets archived in `evidence/blind-output-test/<skill>-<date>.md`. The log is a primary input to the next encoding cycle.

## What the Test does NOT measure

- It does NOT measure whether the script will perform commercially. (Performance is the algorithm's verdict, measured via `/audit-retention` after publish.)
- It does NOT measure factual correctness. (Truth Gate measures that.)
- It does NOT measure brand-safety. (Banned-vocabulary + sponsor-disclosure measure that.)
- It does NOT measure whether the creator agrees with the substance. (That's a separate review pass — the creator can accept that the OS produced a "voice-faithful" script and still reject the substance.)

The Blind Output Test only measures: **can someone tell this came from the system, or could it plausibly have come from the creator?**

---

## Failure response

When a Blind Output Test fails:
1. Archive the test log + reasoning
2. Identify the dominant failure signals from evaluator reasoning
3. Update the relevant compartment fields (most often `creator_identity_matrix.brand_voice_architecture.phrases_to_use` / `phrases_to_avoid` / `language_patterns`)
4. Re-run the skill on a fresh prompt
5. Re-test in 2+ weeks (after enough new outputs to verify the calibration held)

A skill is NOT promoted to production until the Blind Output Test passes.

---

## Cycle-level metric

The OS tracks Blind Output Test pass rate as a top-level encoding-quality metric:

- Cycle 1: typical 0–25% pass rate
- Cycle 2: typical 25–50%
- Cycle 3: typical 50–70%
- Cycle 4–5: typical 70–85%
- Cycle 5+: typical 85%+ (Elite tier)

The pass rate is the operational definition of how complete the encoding is. When pass rate is at 85%+ across all production skills, the encoding is at Elite tier and the OS is producing output indistinguishable from creator's own.
